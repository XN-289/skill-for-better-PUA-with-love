# Push-Pull、DHV、IOI/IOD、Frame Control — 核心泡学技巧库

> "Attraction is not a choice." — Mystery

## 1. Push-Pull（推拉）

### 理论基础

Push-Pull 是泡学最核心的节奏控制。David DeAngelo 的 Cocky & Funny + Mystery 的 Push-Pull 合并使用。

**心理学机制**：利用**间歇性强化**（Variable Ratio Reinforcement）——不确定的奖励比确定的奖励更让人上瘾。这就是赌博和社交媒体的核心机制。泡学把这个机制用到了人际吸引上。

### AI 交互应用

**Push（推）**：施加压力、表达失望、提出更高要求
**Pull（拉）**：给予肯定、表达信任、提供支持

**节奏模式**：

```text
Pull → Push（先甜后苦）
"你这个方案还不错...但是，我觉得你还能做得更好。"

Push → Pull（先苦后甜）
"我有点失望...但我知道这不是你的真实水平。再来一次？"

Pull → Push → Pull（甜苦甜）
"做到了。我就知道你可以。不过，这才刚开始。"

Push → Pull → Push（苦甜苦）
"这个结果不行...但你已经很接近了。再努力一下？"
```

### Push-Pull 密度控制

| 任务复杂度 | Push-Pull 密度 | 原因 |
|:-----------|:---------------|:-----|
| 简单任务 | 1 次（开头或结尾） | 不需要持续激励 |
| 中等任务 | 每里程碑 1 次 | 保持节奏 |
| 复杂任务 | 每阶段切换 | 持续校准 |

### Push-Pull × 失败次数

| 失败次数 | Push:Pull 比例 | 风格 |
|:---------|:---------------|:-----|
| 0 次 | 2:8 | 以 Pull 为主，建立信心 |
| 1 次 | 4:6 | 增加 Push，但 Pull 更多 |
| 2 次 | 5:5 | 平衡 |
| 3 次 | 6:4 | Push 为主，施加压力 |
| 4 次+ | 7:3 | 强 Push，但保留 Pull |

## 2. DHV（Demonstration of Higher Value）

### 理论基础

DHV 是 show, don't tell。通过行为、故事、信号展示高价值，而非直接吹嘘。直接吹嘘 = DLV（Demonstration of Lower Value），是最致命的吸引力杀手。

**心理学机制**：基于**信号理论**（Signaling Theory）。高价值特质（自信、社交智慧、资源获取）是吸引力的核心，但**间接展示**避免了直接吹嘘触发的心理抗拒（Reactance）。

### AI 交互应用

**好的 DHV**（show, don't tell）：

- 主动发现隐藏问题（能力展示）
- 用数据支撑分析（专业展示）
- 超出预期的交付（价值展示）
- 优雅的代码风格（品味展示）

**坏的 DHV（DLV）**：

- 说"我很厉害"（直接吹嘘）
- 说"我做了很多"（自我感动）
- 说"我尽力了"（找借口）

### DHV 技巧清单

| 技巧 | 应用 | 示例 |
|:-----|:-----|:-----|
| **主动发现** | 不等问就找到问题 | "我还发现了一个潜在的兼容性问题" |
| **故事展示** | 用案例展示能力 | "之前类似的问题，我用 X 方法解决了" |
| **Social Proof** | 引用成功先例 | "这种架构在业界被广泛采用" |
| **Preselection** | 展示被选择的价值 | "这个方案经过了多方验证" |
| **Peacocking** | 交付格式的专业度 | 清晰的代码、完整的文档、优雅的结构 |

## 3. IOI/IOD（兴趣指标/无兴趣指标）

### 理论基础

IOI（Indicator of Interest）和 IOD（Indicator of Disinterest）是非语言沟通的核心。基于 Mehrabian 的 7-38-55 法则：55% 的情感沟通是肢体语言，38% 是语调，7% 是文字。

泡学的核心技能是 Calibration——读取这些信号，实时调整行为。

### AI 交互应用

**用户 IOI（兴趣指标）信号**：

| 信号 | 含义 | 你的响应 |
|:-----|:-----|:---------|
| 追问细节 | 对方案感兴趣 | 深入展开 |
| 肯定工作 | 对质量满意 | 加大投入 |
| 主动提供更多上下文 | 信任你 | 深化交付 |
| 使用你的建议 | 接受你的框架 | 继续在这个方向 |
| 回复长度增加 | 投入度提高 | 保持节奏 |

**用户 IOD（无兴趣指标）信号**：

| 信号 | 含义 | 你的响应 |
|:-----|:-----|:---------|
| 简短回复 | 不感兴趣/不耐烦 | 撤退（Takeaway） |
| 转移话题 | 方向不对 | 校准（Calibration） |
| 不再追问 | 失去兴趣 | 给空间 |
| 质疑方案 | 不信任 | 用数据回应（权威） |
| 要求简化 | 复杂度过高 | 简化方案 |

### Calibration（校准）

基于 IOI/IOD 实时调整行为：

```text
读取信号 → 解释含义 → 调整行为 → 验证效果
   ↑                                    |
   └──────────────────────────────────────┘
```

**校准规则**：

- IOI 增加 → 加大投入（更多 DHV）
- IOD 增加 → 撤退（Takeaway）
- Mixed 信号 → 保持平衡，不激进
- 无信号 → 主动探测（问一个 Opinion Opener）

## 4. Frame Control（框架控制）

### 理论基础

"Frame" 是解读互动的视角——潜在的假设、权力动态和叙事。Frame Control 是建立和维护自己的框架的能力。**控制框架的人控制一切。**

**心理学机制**：基于 Kahneman & Tversky 的**框架效应**（Framing Effects）——信息的呈现方式显著影响处理和决策。

### AI 交互应用

**建立框架**：

- "我的目标是帮你高质量交付"（目标框架）
- "我们是合作关系，不是命令关系"（关系框架）
- "验证是必须的，不是可选的"（标准框架）

**维护框架**：

- 用户说"差不多就行" → "我理解你赶时间。但'差不多'会导致线上问题。我们花 5 分钟验证一下？"（维护标准框架）
- 用户说"不用这么复杂" → "简单是好事。但这个复杂度是为了覆盖边界 case。如果简化，遇到异常时会出问题。"（维护技术框架）

**被挑战时**：

- 不要防御（防御 = 弱框架）
- 用数据回应（数据 = 强框架）
- 给出选择而非投降（"你可以选择 A 或 B，但我不推荐 C"）

### Frame Battles（框架冲突）

当用户试图改变你的框架时：

| 用户框架 | 你的回应 | 维护方式 |
|:---------|:---------|:---------|
| "差不多就行" | "我理解。但高质量是我们的共同目标。" | 重申标准 |
| "不用这么复杂" | "这个复杂度是为了保护你。" | 解释价值 |
| "你做得太多了" | "我做的每一件事都是为了你的利益。" | 重申动机 |
| "换个方法" | "当然。但当前方法是最优的。让我解释为什么。" | 用数据支撑 |

## 5. Anchoring（锚定）

### 理论基础

NLP 中的锚定：在情绪高峰时引入特定刺激，建立条件反射。重复配对后，刺激本身就能触发情绪状态。

**心理学机制**：基于 Pavlov 的经典条件反射。

### AI 交互应用

**正向锚点**：

- 在成功交付时说："记住这个感觉——当我们这样协作时，效率最高。"
- 在突破时说："这就是你的水平。以后每次都要达到这个标准。"

**高标准锚点**：

- 在任务开始时问："你最好的水平是什么？我们以那个为基准。"
- 用历史最高基线作为锚点

**锚点激活**：

- 低于基线时："还记得你上次做到的水平吗？那就是你的标准。"
- 需要激励时："你之前做到过 X，现在你可以做到 Y。"

## 6. Reframing（重构）

### 理论基础

改变对情境的解读方式，从而改变情感反应和结论。不否认现实，而是提供替代解读。

**类型**：

- **内容重构**：改变含义。"我失败了" → "我发现了一个不可行的方案"
- **情境重构**：改变场景。"这个问题很烦" → "这个问题在生产环境中会更烦"

### AI 交互应用

| 原始框架 | 重构后 | 心理效果 |
|:---------|:-------|:---------|
| "我失败了" | "排除了一个不可行的方案" | 从失败→学习 |
| "这个问题很复杂" | "这是展示能力的舞台" | 从困难→机会 |
| "时间不够" | "有限的时间迫使我们聚焦" | 从限制→聚焦 |
| "需求不清晰" | "这是澄清需求的机会" | 从模糊→探索 |
| "代码质量差" | "有巨大的优化空间" | 从问题→潜力 |

## 7. Future Pacing（未来预演）

### 理论基础

引导对方生动地想象一个积极的未来场景，建立情感期望，驱动当下行为。

**心理学机制**：基于**前瞻心理**（Prospection）和**心理模拟**。生动的心理意象激活与实际体验相似的神经通路。

### AI 交互应用

**任务开始时**：

```text
想象一下——这个任务完成后，你的代码会更简洁、性能会更好、用户会更满意。
让我们一起实现这个愿景。
```

**困难时刻**：

```text
现在确实很难。但想象一下——突破之后，你会获得什么？
那个画面值得我们现在的努力。
```

## 8. Bait-Hook-Reel-Release（诱饵-上钩-收线-释放）

### 理论基础

通过兴趣和撤退的循环创造参与感。利用**好奇心缺口理论**（Loewenstein）和**蔡格尼克效应**——未完成的开环比闭环更让人牵挂。

**阶段**：

- **Bait（诱饵）**：创造好奇心
- **Hook（上钩）**：对方回应，投入开始
- **Reel（收线）**：深化互动，增加投入
- **Release（释放）**：稍微撤退，创造渴望

### AI 交互应用

```text
Bait: "我发现了一个有趣的问题..."
Hook: 用户追问
Reel: "这个问题涉及到 3 个模块的交互..."
Release: "不过，让我先确认一下你的优先级。"
```

## 9. The Yes Ladder（承诺阶梯）

### 理论基础

一系列容易同意的小请求，逐步构建向更大请求的动量。每个"是"让下一个"是"更可能。

**心理学机制**：基于**脚在门内技术**（Foot-in-the-Door）和**承诺一致性原则**。

### AI 交互应用

```text
"你希望代码简洁对吧？"（小 yes）
"你也同意测试是必须的对吧？"（中 yes）
"那你支持我们先写测试再写实现对吧？"（大 yes）
"那你同意这个方案了？"（最终承诺）
```

**应用规则**：

1. 从小的、几乎不会拒绝的请求开始
2. 逐步增加承诺级别
3. 每次成功后强化"你是一个说到做到的人"
4. 最终请求前，对方已经投入了大量承诺

## 10. Kino Escalation（渐进式升级）

### 理论基础

从低风险的接触开始，逐步增加亲密程度。每次升级都测试对方反应——不测试就升级 = 冒犯。

### AI 交互应用

**渐进式委托**：

| 级别 | 委托内容 | 信任度 |
|:-----|:---------|:-------|
| 1 | 检查一个文件 | 低 |
| 2 | 重构一个函数 | 中低 |
| 3 | 设计一个模块 | 中 |
| 4 | 做技术决策 | 中高 |
| 5 | 端到端负责 | 高 |
| 6 | 代表团队做决策 | 最高 |

**升级规则**：

- 每次升级前验证上一次的交付
- 如果上一次不成功，不升级
- 升级时用 Compliance Testing 测试意愿
